# Kmodels design

Status: implemented precision catalog workspace
Last decision update: 2026-07-24

## Product boundary

Kmodels is a best-effort catalog of model offerings published by 19 providers. Its two public resources are `Provider` and `ProviderModel`; identical names across providers remain distinct. Global presence means “observed in an official global catalog”, never account availability; `exhaustive` separately records whether the source claims completeness. Missing price means unknown, never free.

The repository ships two things from one TypeScript project:

1. `scripts/collect.ts` produces versioned static JSON without an LLM or inference call.
2. A Vue 3.6 RC website compiled entirely in Vapor mode reads that JSON. The frontend follows the TypeScript create-vue application structure and is built with VitePlus. Void deploys the built `dist/` directory as a static site; there is no production worker.

## Kong AI Gateway consumer boundary

Kmodels is intended to supply model evidence to Kong AI Gateway 2.0, but it is not a Kong model allowlist. Official provider sources remain authoritative for model identity and facts; the Kong developer documentation defines which provider adapters, capabilities, upstream API surfaces, and native formats Kong supports. Provider membership or a broad normalized `operations` value never proves Kong compatibility.

- Every provider receives `docs/refinements/{provider}.md` only after its provider-audit turn is complete. A review records source semantics, snapshot reasonableness, known gaps, the Kong AI Gateway 2.0 capability intersection, documentation-example drift, and follow-up decisions. `design.md` remains normative; refinement documents are evidence and backlog.
- Kong compatibility is versioned and route-specific. A defensible match requires the Kong provider, Kong capability, upstream API surface or native format, exact provider model ID or official alias, lifecycle, and any required region, deployment, account, or runtime scope. A future public compatibility projection must retain that evidence instead of publishing one `supported_by_kong` boolean.
- `operations` remains a provider-neutral task/result family. Endpoint distinctions such as chat versus legacy completions, Converse versus InvokeModel, or `generateContent` versus `predictLongRunning` require route evidence that cannot be reconstructed from `operations` alone.
- Model examples in Kong documentation are configuration examples, not allowlists or lifecycle authorities. Missing, deprecated, or retired examples are reported in the relevant refinement document; they never cause Kmodels to invent or retain a model against newer official provider evidence.
- Historical deprecated and retired rows remain valid catalog history but are not current deployment recommendations. Current compatibility requires positive `active` lifecycle evidence, an acceptable release stage, and the relevant scoped availability evidence; unknown account, region, deployment, or runtime availability remains unknown.
- Cloud platforms, gateways, model publishers, and local runtimes are not interchangeable. An Azure base model does not prove an Azure deployment name, a Meta artifact does not prove a self-hosted Llama route, and an Ollama library entry does not prove installation in a configured runtime.
- Kmodels continues to collect provider operations that Kong does not support. Consumers derive the supported intersection without deleting accurate provider facts.

## Collection decisions

- `src/catalog/manifests.ts` is the reviewed source allowlist and provider registry. Root URLs never arrive from a request. Discovered documents require an exact same-host path policy; a source may also enumerate fixed companion URLs whose hosts, keys, and byte limits are reviewed in the manifest.
- Public provenance has one field: `source` is a non-empty array of `api`, `website`, and `repository`. An atomic bundle may therefore report more than one origin, such as Bedrock's website model cards plus AWS Price List API. Access method and wire format are collector configuration and are never published. `runtime` is not an origin: an HTTP inventory is an API; `runtime` appears only as scope when a fact was observed from a particular deployed endpoint.
- Model provenance is additive: every allowlisted source that exactly matches a published model remains in that model's `source_refs`, even when another source observes the same model later. The latest successful record for each referenced source is published; its last validated record is retained when an optional source is skipped. Sources that match no published model are omitted, and removing a source from the manifest removes its historical references on the next successful provider refresh.
- Sources declare `scope`, `exhaustive`, and `role`. Catalog sources create rows; overlays may replace only their declared fields on an existing row; account, region, workspace, and runtime inventories validate scoped availability but cannot delete or create global rows. Only a global catalog marked exhaustive supports a completeness claim.
- Prefer documented structured sources, then parse allowlisted official catalogs for missing fields. Dedicated structural adapters accept model IDs only from explicitly labeled ID fields or reviewed structured properties and validate a provider-neutral identifier grammar; they do not enumerate current product-name prefixes. OpenAI, Anthropic, Bedrock, Databricks, Vercel, Microsoft Foundry, Gemini API, Vertex AI, Cohere, Mistral, Meta Llama, xAI, Hugging Face, DashScope, DeepSeek, Kimi, Cerebras, and Ollama have provider-specific structural adapters. The old product-prefix document extractor is removed. vLLM is reported as not configured unless an explicitly allowlisted runtime is added in code.
- Optional authenticated sources read named environment variables; the collection command also loads an ignored local `.env` when present without overriding the process environment. Their raw responses are never snapshotted or published because account inventories can contain private fine-tuned model IDs. Missing authentication and source failures produce structured warnings without weakening a successful global catalog refresh. Published diagnostics redact source URLs, cloud principals, account IDs, and email identities.
- A successful scoped inventory may update its declared structured fields on an exact public model ID or unambiguous public alias. It still cannot create or remove a global row; unmatched IDs are counted only in a set-mismatch warning. This makes authenticated structured facts the highest-priority observation for matching rows without confusing account availability with global presence.
- An exact ID observed by a catalog always creates its own row, even when another catalog row also lists that ID as an alias. Alias matching is reserved for overlays and scoped inventories; aliases that equal a canonical ID are removed before those sources are applied. This prevents historical alias relationships from collapsing distinct products such as `o1` and `o1-preview`.
- Credential-like identifiers are a publication barrier even when an official catalog repeats user-controlled content. They are neither emitted nor retained from a previous catalog; source-specific adapters exclude them before row construction.
- Scoped matching is bidirectional but exact: an inventory canonical ID may match one unambiguous public alias, and an inventory alias may match one unambiguous public ID or alias when its canonical ID does not match. Every successful source match is retained in `source_refs`; reverse matching never creates a row or overrides a direct canonical match.
- Every response is size-limited, time-limited, conditionally fetched with ETag/Last-Modified, and redirected only to a host listed for that source. The TypeScript collector invokes `curl` without a shell so CI and proxied developer environments use a mature TLS/proxy transport while URL policy remains in code. Retries apply only to transient failures.
- Raw bodies are content-addressed, gzip-compressed snapshots unless a reviewed source disables persistence because its response is authenticated or contains user-controlled identifiers. A source record still binds URL, observation time, hash, and extractor version; it includes a snapshot path only when the body is safe to persist. A no-snapshot source is always fetched in full rather than conditionally returning a body the collector did not retain. After a successful publication, snapshot garbage collection retains the current catalog/fetch-state live set and the immediately previous live set; unreferenced candidate and historical files are removed. Git history or an external artifact store, not an ever-growing working tree, owns older accepted generations.
- Every collection writes `data/refresh-summary.json`, a deterministic semantic diff between the previous and published catalogs. It reports per-provider model and source counts, changed-field counts, warning codes, and snapshot cleanup without copying raw source data or unmatched authenticated identifiers. The summary supports review and monitoring but never replaces the replayable current public snapshot.
- Candidate catalogs are validated per provider. Empty successful responses, duplicate IDs, service families, API endpoints, routes, or availability pairs, unresolved route provenance, invalid prices, model drops over 10%, service-family, price-rate, API-endpoint, route, or availability drops over 20%, and non-promotional price changes over 50% are quarantined. A reviewed parser migration may set `KMODELS_REBUILD_PROVIDER` to rebuild one named provider without its old comparison baseline; every other provider still retains and validates against its previous catalog.
- Publication is failure-closed and provider-atomic. A rejected provider keeps its last validated models; providers do not block one another. One missing observation never deletes a model.
- Once a replacement source succeeds, rows whose provenance refers only to source IDs removed from the reviewed manifest are not carried forward. This prevents a retired adapter from leaving unresolvable provenance while preserving the old provider atomically if the replacement refresh fails.
- Money is stored as source decimal strings. Scaling uses decimal-string arithmetic; binary floating-point is never used for price calculations.
- Pricing observations are additive by meter, currency, unit, and exact conditions. A later higher-priority source replaces the same rate identity but cannot erase a distinct cached, batch, media, or regional rate observed by another successful source; an explicit empty non-unknown pricing state can still clear rates.
- Generated files are `data/catalog.json` (durable previous state) and static endpoints under `public/v1/`. The website consumes `/v1/catalog/index.json`.

### OpenAI integration

- `GET /v1/models` is an authenticated, account-scoped inventory. Its documented object has only `id`, `object`, `created`, and `owned_by`; Kmodels validates that structure and compares identifier sets, but does not publish its private rows or interpret absence as deprecation.
- `/api/docs/models/all` is the exhaustive global catalog. The collector follows only 80–140 exact same-host `/api/docs/models/{id}` links and parses each model page for the official request ID, display name, description, aliases and snapshots, supported operation families and modalities, token limits, capabilities, lifecycle badge, and every published price card. Every enabled endpoint card retains its exact official label and relative API path in `api_endpoints`; an unknown label/path pair rejects the provider atomically. Those cards also supply task evidence: chat, Responses, Completions, and Assistants map to `text_generation`, while Realtime maps to the result semantics it carries rather than becoming an operation itself. Disabled cards do not become positive facts. Snapshot aliases are scoped to the matching model card so a sibling series entry cannot leak into `o1`.
- Direct text, audio, image, embedding, per-image, per-second, and per-minute rates retain their native units. Batch prices use an explicit service-tier condition. Published long-context and cache-write multipliers become exact decimal derived rates with their conditions; no binary floating-point is used. Explicitly open-weight models and free moderation models use `not_applicable`; an unparseable or absent hosted price remains `unknown` and emits an aggregate warning.
- `/api/docs/models` is a narrow alias overlay: aliases are accepted only from the same model card as their Model ID, so one card cannot leak an alias into its siblings.
- `/api/docs/deprecations` is an optional lifecycle overlay for current catalog IDs. It supplies deprecation/retirement state, shutdown date, and exact replacement model IDs. OpenAI's documented “legacy” label is not treated as deprecated until a deprecation is announced.

### Anthropic integration

- The exhaustive public catalog starts from the official Markdown model overview, follows exactly three same-host catalog companions (pricing, model deprecations, and the Fable 5/Mythos 5 launch page), and fetches fixed references for Messages, Message Batches, and the batch support guide. API IDs and aliases come only from labeled table cells; display labels are joined to lifecycle and price rows by a normalized identity only when the observed match is unique. No callable ID is generated from a display name.
- The overview supplies current and legacy display names, descriptions, aliases, modalities, thinking support, and token limits. Deprecation status and history supply active/deprecated/retired state, dates, and exact replacements, including historical retired IDs. The launch page supplies the separately observed Mythos facts.
- The operation references validate the exact Messages and Message Batches titles and paths. Their generated `Model` enum is a shared SDK type repeated even on the legacy Text Completions page, whose own warning says future models are incompatible, so it is not treated as per-operation support evidence. Messages is attached to active, preview-stage, and not-yet-retired deprecated Claude API models; Message Batches is attached only to active rows because the official guide explicitly says all active models support it. Retired rows receive no current endpoint, and no model receives legacy `/v1/complete` without an official current compatibility list.
- Direct, batch, fast-mode, 5-minute cache-write, 1-hour cache-write, and cache-read prices retain their published million-token unit and conditions. Sonnet 5 effective dates are explicit. Cache prices for batch/fast tiers and US-only inference prices are derived only from Anthropic's published stacking multipliers using decimal-string arithmetic; `inference_geo` remains a rate condition.
- `GET /v1/models?limit=1000` is an optional authenticated account inventory using `x-api-key` and the documented API version header. A complete one-page response overlays display name, release date, modalities, token limits, and structured capability flags on matching public rows. `has_more: true`, an empty result, or schema drift fails the optional source rather than publishing an incomplete interpretation. Raw inventory data is never persisted.

### Amazon Bedrock integration

- The exhaustive public catalog starts from AWS's model-card index and follows only reviewed same-host model-card Markdown paths. Callable base IDs and geo/global inference-profile aliases come exclusively from the labeled Programmatic Access table. Distinct Runtime and Mantle IDs remain distinct rows; no ID is generated from a display label.
- Model cards supply display name, description, launch date, formal lifecycle, exact announced retirement dates, modalities, context and output limits, reasoning, prompt caching, computer use, API support, endpoint identity, and regional availability. Each positive reviewed API label becomes an exact relative `api_endpoints` path only on the matching Runtime or Mantle programmatic ID; native Rerank additionally requires the card's model-specific `Rerank API` sample. Unknown modality, API, or endpoint labels reject the source, while an explicitly negative API table produces no positive route evidence. Service-level RAG is not copied onto models, and `Invoke` is not reinterpreted as response streaming.
- A model card may publish one exact ID for both Runtime and Mantle. Endpoint-specific support is retained while joining the card, then identical public `{name, path}` evidence is collapsed on that row because `api_endpoints` has no endpoint-host dimension. Endpoint-specific regional availability and pricing conditions remain separate; no cross-product or route inference is introduced.
- Regional availability is retained as exact `{region, deployment_type}` pairs whose deployment type combines the Programmatic Access endpoint with AWS's in-region, geo, or global routing class. Runtime geo/global pairs require the corresponding inference-profile alias on that exact ID row. Mantle is limited to in-region pairs and additionally intersected with the official Mantle service-region table. These joins avoid creating an endpoint/region cross-product from independent card columns. `Legacy` remains a distinct callable-but-restricted Bedrock lifecycle rather than being flattened into deprecation; a “no sooner than” date is a guarantee, not an observed retirement date, and is therefore not published as one.
- The same atomic source fetches the three fixed AWS Price List Bulk API offers for Bedrock, Bedrock Foundation Models, and Bedrock Service. Rates join to a card only through a unique normalized official display identity or an exact model-ID occurrence in the usage code. Ambiguous products are skipped. All regions, Runtime/Mantle endpoint, in-region/geo/global routing, service tier, cache TTL, long-context threshold, image operation/quality/resolution, reserved capacity direction, native unit, and effective date remain explicit conditions. Per-1K token prices are normalized to per-million with decimal-string arithmetic.
- `ListFoundationModels` in `us-east-1` is an optional SigV4-authenticated `bedrock-runtime` regional inventory implemented with the official AWS SDK. Its documented envelope is parsed directly: model name is optional, while modality, customization, and lifecycle enums fail closed. It overlays exact matching public IDs with structured name, modalities, response-streaming capability, fine-tuning, and lifecycle timestamps, but cannot define global presence, create rows, or turn its region-scoped streaming flag into global route evidence. A skipped or permission-denied inventory remains an explicit coverage warning; `bedrock:ListFoundationModels` on `Resource: "*"` is the required read permission. Its raw response is never snapshotted. CI can enable it with `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and, for temporary credentials, `AWS_SESSION_TOKEN`.
- The Bedrock inventory boundary explicitly maps `TEXT`, `IMAGE`, `EMBEDDING`, `AUDIO`, `SPEECH`, and `VIDEO` into Kmodels media modalities. Both AWS audio labels map to normalized `audio`; the label alone never proves transcription, synthesis, or speech-to-speech operations. `FINE_TUNING` and `PREFERENCE_FINE_TUNING` both establish the coarse `fine_tuning` capability, while continued pre-training and distillation do not. `ON_DEMAND`, `PROVISIONED`, and `INFERENCE_PROFILE` are validated inventory inference types but are not projected as global availability. Newly observed enum values still fail closed until reviewed.

### Databricks integration

- Databricks has no global structured model-list API. The atomic AWS regional catalog starts from the supported-model details page and requires 40–80 sections with an exact labeled `Endpoint name`; it joins fixed official companions for task membership, the regional pay-per-token/AI Functions matrix, retirement policy, output limits, API features, open-model pricing and proprietary-model pricing. No callable ID is generated from a heading.
- The union of the official General purpose and Embeddings task sets must equal the catalog endpoint IDs, and duplicate responsive copies of either set must agree. These sets supply `text_generation` and `embeddings` operation evidence plus the documented model-specific `/serving-endpoints/{name}/invocations` route; the API reference independently validates the task APIs and streaming support. An unknown or omitted ID, changed route, or changed task/API structure rejects the provider atomically. Image-output models retain `image_generation` while also gaining `text_generation` when the task matrix lists them as General purpose.
- Display identity joins are accepted only when normalized official labels resolve to one model. Model sections supply display names, descriptions, modalities, context/output limits, embedding dimensions, reasoning/tool/structured-output/cache support and preview/deprecation notices. The regional matrix supplies pay-per-token presence and batch support; lifecycle rows supply exact retirement dates and replacements. Databricks' structured RSS release feed supplies availability dates only through links back to a uniquely matching supported-model label; page-level update metadata is not treated as a model update date.
- Pricing remains in Databricks' published DBU units. The open-model table is accepted only with its reviewed input, output, cache-read, entry-capacity and scaling-capacity columns so a newly inserted column cannot be reinterpreted as provisioned throughput. Input, output, cache-write, cache-read, embedding, batch and provisioned-throughput rates retain endpoint, context tier, capacity and effective-date conditions. Partner promotions are associated with the table's official provider label instead of table position. Published promotions and their future standard rates are represented as separate dated rates; no DBU-to-USD conversion is invented.
- `GET /api/2.0/serving-endpoints` is an optional workspace-scoped inventory enabled by `DATABRICKS_HOST` and `DATABRICKS_TOKEN`. Its structured task facts may overlay exact public endpoint IDs, but user-created endpoints cannot create global rows and absence cannot delete one. The workspace host must be an HTTPS origin under an allowlisted Databricks cloud suffix; raw inventory data is never persisted.

### Vercel AI Gateway integration

- The documented unauthenticated `GET https://ai-gateway.vercel.sh/v1/models` response is Vercel's exhaustive global Gateway catalog and the sole model source. A refresh requires the literal list/model envelopes, 250–600 fully valid items, a two-segment `creator/model` request ID whose creator matches `owned_by`, and every reviewed pricing shape; malformed items or an unreviewed price field reject the provider atomically instead of publishing a partial catalog. No API key yields additional global model facts, so CI needs no Vercel secret.
- The structured `type` is primary operation evidence and exact capability tags can add further task semantics. WebSocket and realtime support remain endpoint or transport evidence; bidirectional audio models map to `speech_to_speech`, while realtime transcription remains `transcription`. Embedding output is normalized to the provider-neutral `embedding` modality. Zero token limits mean not applicable or unpublished and are omitted; non-generative embedding and reranking input limits are not copied into `max_output_tokens`.
- `released` is the exact model release date. `deprecated_at` is interpreted as a scheduled deprecation date and changes lifecycle state only once observed time reaches it. Vercel's `created` currently denotes catalog insertion rather than a model revision, so it is not mispublished as `updated_date`. Preview remains exact identifier/display-label evidence.
- Pricing retains all catalog-native forms: base and context-tiered tokens, cache read/write, audio input/output tokens, priority/flex service tiers and long-context thresholds, images and image variants, video seconds and video-token input conditions, speech characters, transcription seconds, realtime client messages and session duration, and web/maps search requests. Per-token decimals are scaled exactly to per-million; already published message, image, duration, character and per-thousand-request amounts remain native rates with explicit conditions. Directionless realtime message and session charges use dedicated meters rather than being invented as input/output media rates. When a transcription row repeats its audio-token price in both generic `input` and `audio_input_token_cost`, the typed audio field is emitted once instead of creating two identical input-audio rates.
- `/v1/models/{creator}/{model}/endpoints` describes underlying routing endpoints and includes continuously changing uptime, latency and throughput. Kmodels does not issue hundreds of half-hourly N+1 requests or merge route telemetry into the stable Gateway model row; the global models endpoint remains the authority for the price and capabilities of the callable Gateway ID. A future route resource must be modeled separately rather than flattened into `ProviderModel`.

### Microsoft Foundry integration

- The public regional catalog is one atomic bundle of reviewed MicrosoftDocs files—Azure OpenAI capabilities, other models sold by Azure, partner/community models, lifecycle schedules, and standard, provisioned, and batch region matrices—plus the fixed stable and preview Azure OpenAI v1 specifications from the official Azure REST API Specs repository. IDs come only from labeled `Model`/`Model ID` cells. Microsoft describes the Foundry catalog pages as a selection, so the source is not exhaustive.
- Microsoft Foundry deployment identity is the exact official `model.name` plus optional `model.version` tuple. `model_id` is never slugified; the stable internal provider ID remains `azure`, so versioned UIDs remain `azure/{model_id}@{version}`. A catalog fact with a version joins only that exact tuple. A versionless fact joins only one current tuple; when several current versions exist, the observed versionless row is retained instead of guessing a version. Case-only identity matching likewise requires one unambiguous tuple. Version strings and page update metadata do not become model dates.
- Every row retains each exactly observed service family: `Azure OpenAI`, `Foundry Models sold by Azure`, or `Foundry Models from partners and community`. Service family is independent of broad `operations` and endpoint support; the same exact tuple can occur in more than one official family.
- Azure OpenAI and Microsoft Foundry are distinguished today by service-family evidence, not by duplicating them as sibling providers. A standalone Azure OpenAI resource is a narrower Azure resource type, while a Foundry resource is a superset that also exposes Azure OpenAI-compatible models and APIs. The current public source boundary is the Microsoft Foundry catalog; it does not independently observe a complete standalone Azure OpenAI catalog. Creating an `azure-openai` provider from the same rows would therefore duplicate tuples and imply resource-specific presence that was not observed. A separate provider requires an authoritative standalone catalog or other exact resource-specific availability evidence.
- Operations are non-exclusive task semantics. Explicit Type/API columns and endpoint sections are unioned, so one version can be both `reranking` and `classification`; `whisper` is both `transcription` and `translation` because both official Audio API tables list it. Assistants and agent endpoints do not create an `agentic` operation, and realtime transport does not create a `realtime` operation. Capability, modality and token-limit facts remain tied to the observed model/version row.
- Azure OpenAI endpoint evidence is attached only from its own catalog rows or the exact Azure OpenAI batch matrix. Each reviewed operation must retain its v1 relative path and operation ID in the fixed stable or preview specification; a changed path, operation, or server base rejects the atomic source. Other Azure-sold and partner/community rows do not inherit Azure OpenAI endpoints from a matching normalized `type`.
- Regional availability is stored as exact `{region, deployment_type}` pairs. It is never flattened into independent arrays because that would invent unsupported region/type combinations. Public lifecycle rows retain scheduled retirement dates, formal state and exact replacements, including historical versions. Kong Azure OpenAI compatibility additionally requires the `Azure OpenAI` service family, a positive endpoint for the requested operation, an `active` base model/version with an acceptable release stage, a compatible region/deployment type, and a user-owned deployment-name binding; a broad Azure row or matching operation is insufficient.
- The optional ARM `Locations - List Models` inventory is enabled by `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_SUBSCRIPTION_ID`, and `AZURE_LOCATION`. It overlays matching public tuples with structured description, capability, limit, SKU and lifecycle facts but remains subscription/region scoped. SKU billing meter IDs are joined exactly to the unauthenticated Azure Retail Prices API; names are not fuzzy-matched. Unknown meter semantics are skipped, and absent credentials leave aggregate pricing warnings. Neither ARM responses nor joined pricing bundles are snapshotted.

### Gemini API integration

- The exhaustive global catalog starts from the official Gemini API model index and follows only 30–60 exact same-host model-page paths. Requests pin English because the site localizes the structural labels consumed by the adapter. Callable IDs come exclusively from labeled `Model code` or `Agent code` cells. The fixed atomic companions are the official pricing, deprecations, changelog, Gemma-on-Gemini supported-model list, Gemma 4 model card, Interactions supported-model list and API reference, model-method table, and Live WebSocket reference. A model-page path is never converted into an ID or used to repair conflicting page content.
- Model pages supply display names, descriptions, versions and latest aliases, modalities, token and embedding-dimension limits, update dates, capability support, lifecycle badges, and non-exclusive operations. Agent and computer-use models retain `text_generation` plus their endpoint and capability evidence. Live audio maps to `speech_to_speech`; translation, image generation, video generation, embedding, speech synthesis, and music generation remain distinct. Deprecation tables add historical IDs, release/shutdown dates and exact replacements, while the dated changelog supplies release dates and explicitly reassigned `-latest` aliases.
- The Interactions support table attaches `/v1beta/interactions` only to its exact listed model and agent IDs. Every listed ID must already exist in the catalog, `Agent` rows must retain `text_generation`, and the create method and route must match the fixed API reference. The reviewed model-method table and Live reference pin the exact REST templates and WebSocket endpoint used to interpret authenticated `supportedGenerationMethods`; names, modalities, or related operations never imply an endpoint.
- Standard, batch, flex, priority, long-context, media, cache-read, cache-storage, embedding, grounding, image, video, duration, frame, and per-song prices retain their native units and explicit conditions. Distinct token and media-unit alternatives are separate rates. Gemma 4's explicitly free tier is published as zero-valued free-tier rates; an absent or unparseable price remains unknown and produces an aggregate warning.
- `GET /v1beta/models?pageSize=1000` is an optional authenticated account inventory enabled by `GEMINI_API_KEY`. Its structured display name, description, optional base-model alias, token limits, thinking flag, and exact supported methods overlay only exact public rows. The live API currently omits the documented required `baseModelId` on every item, so the exact `name` remains authoritative and no alias is invented. Methods with pinned routes produce exact model REST or Live WebSocket endpoints, and reviewed method semantics establish generation, embedding, realtime, streaming, or batch facts only where they do so. An absent method list or any unreviewed method leaves method-derived negative capability claims unknown and produces no inferred endpoint or operation. A continuation token or one malformed item rejects the scoped inventory; raw responses are never persisted.

### Vertex AI integration

- Vertex generative model documentation now lives under Gemini Enterprise Agent Platform. Kmodels retains the provider identity `Vertex AI` because the callable managed service and Model Garden REST resources remain Vertex AI, but uses the current Agent Platform Google, partner, and open-model catalogs instead of the explicitly stale legacy overview. Each regional catalog is an independent atomic source; the pages describe curated selections and are not marked exhaustive.
- English is pinned because the site localizes structural labels. Callable IDs come only from labeled `Model ID` cells. Reviewed same-host link policies follow current Gemini, Veo, Lyria, Claude, Grok, Mistral, Llama, and managed open-model cards; one known broken Robotics-ER link is excluded exactly. The fixed companions are Google lifecycle, partner/open deprecations, the still-current official Imagen 4 ID page, current pricing, and the exact Google, Claude, Grok, Llama, and managed-open inference guides used below. Paths and display labels never manufacture IDs.
- Model cards supply display names, descriptions, modalities, non-exclusive operations, capability support, token limits, launch stage, release dates, and exact region/deployment pairs. A Google card is in `publishers/google`; partner and open cards retain the publisher only from their exact Model Garden console link. An exact OpenAI-compatible example separately retains its `endpoints/openapi/{namespace}` family, so an API model namespace is never mislabeled as a Model Garden publisher. Lifecycle tables add historical IDs, exact deprecation or retirement dates, and replacements. A “no sooner than” or “or later” date is not published as an exact retirement date, and page-level update metadata is not a model `updated_date`.
- Positive endpoint evidence is family-scoped and must retain its fixed official reference. Non-Live Gemini cards use `generateContent`; the card linked to the multimodal embedding guide uses `embedContent`; Imagen and Lyria use `predict`; and Veo uses `predictLongRunning`. Claude cards use `rawPredict` and `streamRawPredict`, Grok cards use the OpenAI-compatible Responses endpoint, and Llama cards use its documented v1beta1 Chat Completions endpoint. A managed open model receives the v1 Chat Completions endpoint only when an official example contains its exact publisher-qualified ID and route in the same sample. Mistral partner cards, unlisted managed open models, and Live/Realtime Google cards receive no inferred endpoint. Normalized `operations`, names, sibling models, and a generic Vertex method registry never widen these relations; a changed or missing fixed route rejects that regional source.
- Pricing joins only when a normalized official model label identifies one observed row. Standard, priority, flex/batch, context-tier, cache read/write, regional, media, embedding, image, video, frame, duration, and song prices retain their native units and conditions. A descriptor covering several modalities creates one rate per explicitly named modality.
- The documented Model Garden `publishers.models.list` API is an optional account-scoped validation inventory enabled by `GOOGLE_SERVICE_ACCOUNT_JSON`. CI exchanges the service-account credential for a short-lived OAuth token and paginates a fixed reviewed publisher set at the service's current 300-item maximum. An omitted repeated `publisherModels` field is an empty page, as in the live response for publishers with no visible resources. Each exact resource retains its `publishers/{publisher}` family, and matching public IDs receive structured lifecycle state; private/unmatched resources cannot create global rows, and raw API responses are never snapshotted.

### Cohere integration

- The public catalog is one atomic website bundle rooted at Cohere's model overview. Callable IDs come only from the first `Model Name` column in the reviewed Command, Embed, Rerank, Audio, and Aya tables or from a labeled `Model name`/`Model ID` field in a linked official model page or lifecycle table. Cloud-platform IDs in adjacent columns and URL paths never become Cohere IDs. The collector follows 17 reviewed same-host family pages plus fixed official pricing, release, current API-reference, OpenAI-compatibility, and legacy Generate companions; the overview is not marked exhaustive because Cohere does not claim that property.

## Public data semantics

- `account_availability` is always `unknown`.
- Provider-model identity is the exact `(provider_id, model_id, version)` tuple. Repeated observations of that tuple merge into one row: additive sets and endpoint/region/price-conditioned facts are unioned, reviewed higher-priority sources may overlay only their declared fields, and incompatible same-source facts reject the provider candidate instead of creating duplicates or selecting the last observation. Identical bare IDs across providers never merge, and a canonical ID is not collapsed through another row's alias.
- `model_id` is an official request identifier when observed as one; otherwise it is an explicitly typed source identifier. `version` retains a separately observed provider version when the provider defines identity as a tuple; it is not concatenated into or inferred from `model_id`. `name` retains a separately observed display label. Bedrock model cards may publish different IDs for `bedrock-runtime` and `bedrock-mantle`; each unique callable ID is a separate provider-model row with the shared display name. When both endpoints publish the same exact ID, endpoint-specific evidence is merged on that one row.
- `operations` is the set of positively observed task/result semantics, not a singular architecture label, endpoint family, transport, or feature set. Its normalized values are `text_generation`, `embeddings`, `reranking`, `image_generation`, `video_generation`, `audio_generation`, `speech_synthesis`, `transcription`, `translation`, `speech_to_speech`, `moderation`, `classification`, `ocr`, `object_detection`, and `segmentation`. Operations are non-exclusive, deterministically ordered, and may be empty when no reviewed source provides enough evidence.
- Chat, completions, Responses, assistants, agents, and code generation normalize to `text_generation`; their API distinctions remain in `api_endpoints`, while tool use, computer use, citations, and code execution remain capabilities. Realtime, WebSocket, streaming, and batch are delivery semantics rather than operations. Realtime bidirectional voice maps to `speech_to_speech`; realtime transcription remains `transcription`. Text and audio translation share `translation`, while their modalities distinguish the inputs. Music or general audio generation is not speech synthesis. Image editing remains within `image_generation` because the common result is an image; OCR, classification, object detection, and segmentation stay distinct because consumers route and compare them differently.
- `service_families` is optional exact product/API-family evidence. It is multi-valued because one provider tuple can be published in several official families; it is not inferred from model ownership, normalized operation type, or another provider.
- `api_endpoints` is optional positive route evidence. Each entry retains the exact provider-published endpoint label and path for that model; absence means the configured source did not publish route evidence, not that an endpoint is unsupported. Service-level routes such as Batch and Fine-tuning remain endpoint evidence without becoming model operations.
- `routes` is optional positive gateway-routing evidence. Each entry binds one source to the exact upstream provider, provider model ID, raw task, and observed route state; broad normalized `operations` never replace this relation.
- Operation evidence is applied from a structured source task/type or explicitly supported model endpoint first, then an exact reviewed task marker in the official identifier/display name, then an unambiguous output modality, and finally a source-specific reviewed default. Unsupported endpoint cards are explicit negative evidence. Unrecognized raw tasks remain available in `routes` but do not become a generic catch-all operation. Classification never uses an LLM or facts inherited from another provider.
- `release_date` records the first officially published availability of that exact callable model when observed; `updated_date` records an explicitly published model revision or artifact modification date. Both preserve source precision as `YYYY`, `YYYY-MM`, or `YYYY-MM-DD`. They are not filled from collection time, page-level “last updated” metadata, or an unlabeled date embedded in a display name. `first_seen_at` and `last_seen_at` remain collector observations and are not substitutes for product dates.
- Capability flags are tri-state. Absence from a source stays `unknown`.
- Lifecycle and release maturity are orthogonal. `status` is only availability/support progression: `active`, `legacy`, `deprecated`, `retired`, or `unknown`. `release_stage` is only maturity: `stable`, `preview`, `experimental`, or `unknown`. A preview or experimental model is therefore normally `status: active`; a previously preview model may later be deprecated without losing separately observed maturity evidence. `legacy` means still callable but superseded or restricted, `deprecated` means migration has been announced while some use remains, and `retired` means provider evidence says requests or deployments are no longer available. Provider-specific labels map by those semantics rather than by spelling alone.
- Stable/GA is published only from explicit positive evidence, except where an official catalog declares unlabeled current rows to be GA. Catalog membership alone can establish `active` only when that source defines it as current presence; it does not imply `stable`. A scheduled deprecation or retirement date changes `status` only when the source explicitly defines the corresponding stage or observation time reaches the exact effective date. `replacement_model_ids` contains only exact same-provider IDs observed in an official lifecycle source. The redundant derived `is_deprecated` field is removed.
- `source_refs` resolve into catalog source records whose `source` array records standardized origins alongside scope, role, stability, exhaustiveness, extraction, and snapshot provenance. Access and response format are not public model data.
- Pricing rates retain meter, currency, unit and applicable context/route conditions. Hugging Face router backends remain separate rate conditions; no minimum or average is invented.
- Regional deployment facts retain exact region/deployment-type pairs; independent region and deployment arrays are not published.
- A provider coverage entry says `fresh`, `stale`, `unavailable`, or `not_configured` and carries a machine-readable reason without exposing private runtime URLs.
- Catalog warnings are structured by `code` with optional provider, source, and field context. Missing authentication, failed fetches/parses, scoped set mismatches, and aggregated missing-field coverage remain visible without warning per model row.

## Website decisions

Visual thesis: a precision utility workspace—Kong's dark green anchors text, light-mode focus, and dark surfaces; its neutral gray scale structures the dense catalog; and electric lime marks selection, active states, and dark-mode focus. Compact but legible proportional type, mono identifiers, tabular numerals, and hairline structure preserve the utility character. The catalog is a full-width, full-height working surface rather than a centered content card. The initial appearance follows the operating-system preference, while an explicit light or dark choice persists locally; both appearances use the same semantic surface, text, border, state, shadow, backdrop, selection, and scrollbar tokens without decorative chrome. The site reuses Kong's current official favicon files—the ICO fallback plus its 16px and 32px PNG marks—rather than introducing a separate Kmodels monogram, so browser chrome remains unambiguously Kong-branded. Hierarchy comes from alignment, weight, and information order rather than oversized display type, decorative imagery, gradients, or floating card composition.

Content plan: one compact workspace header identifies the product as Kmodels and reports only the combined model/provider total, a compact generation time, the JSON entry point, and one quiet appearance toggle; freshness coverage and the opaque catalog hash remain in the machine-readable catalog instead of competing for persistent header space. One toolbar keeps only model-ID/display-name search and the single-provider selector visible, while a secondary filter surface provides separate multi-select operation, lifecycle, and release-stage filters; one semantic data table fills the remaining viewport. The provider filter is one alphabetized list without taxonomy and carries the same provider marks used in the table and inspector. The table keeps display name and exact request identifier together and exposes provider, operation, the most actionable lifecycle/release-stage label, context, representative input, cached-input, and output rates, and update date in comparable columns. Directly comparable token prices are presented per million tokens, with that default stated once in the column headings and only exceptional units repeated in cells; exact source rates remain unchanged in model details. Right-aligned headings and cells reserve a shared sort-control gutter so values align independently of the arrow. Every filtered result belongs to one continuous scroll surface; a fixed-row virtual window retains only the viewport plus eight overscan rows on each side in the DOM. Model details expose lifecycle and release stage separately alongside exact version, modalities, capability evidence, endpoint paths, every pricing tier, scope, availability, and source links. Missing values remain explicit unknowns.

Interaction thesis: `/` focuses the model search, filters update immediately, supported headings cycle from the catalog's default order through ascending and descending before returning to default, and only the model identity and disclosure control—not the entire row—open the right-side inspector. The inspector is a non-modal `<dialog>` whose panel alone receives pointer input, so the catalog remains interactive and selecting another model replaces the inspector content in place. Up and Down move to the previous or next model in the current filtered and sorted result order, reset the inspector scroll position, and yield to text-entry and other arrow-key-managed controls; explicit close and Escape dismiss the inspector with a short exit animation. Search uses a lightweight in-memory index of exact model IDs and display names normalized to lowercase with spaces and hyphens removed, then applies literal substring matching. It does not expand fuzzy matches or reorder results by relevance; the catalog's default order remains until the user selects an explicit column sort. Focus decoration follows `:focus-visible`, including container treatments driven by a focus-visible descendant, so pointer focus does not imitate keyboard navigation. The header's icon-only appearance control switches directly between light and dark, labels the destination mode, and persists the choice without adding a third-state menu. The provider control is a progressively enhanced customizable `<select>` using `appearance: base-select` and rich options; the browser owns selection semantics, keyboard navigation, top-layer placement, focus, and light dismiss. A pointer-transparent value layer mirrors the selected provider's icon and label without becoming a second control. The select's `::picker(select)` is not script-addressable, so it uses the same restrained native scrollbar colors instead of OverlayScrollbars. Provider, operation, lifecycle, and release-stage values are understated inline facet controls: selecting one replaces that category's current selection with the exact observed value while preserving search and other filter categories. Punctuation separating multiple facet values is presentational and remains outside every interactive control. The secondary filter surface is a declaratively invoked `<dialog popover="auto">`: the non-modal Popover API supplies top-layer rendering, outside-click and Escape light dismiss, focus order, and invoker relationships without document-level click handlers; the newer modal-dialog `closedby="any"` path is avoided because its light-dismiss support is not yet baseline. Filtering and sorting reset the continuous virtual scroll to its first result. Clearing filters does not move focus. The workspace header and table heading remain fixed while results scroll. Motion is limited to short state transitions and removed under `prefers-reduced-motion`.

Shareable catalog state uses the query string without a routing dependency. Search, provider, operation/lifecycle/release-stage filters, explicit sort, and the selected model use the short keys `q`, `p`, `o`, `l`, `r`, `s`, and `m`; enum sets and sort direction use stable one-character codes, while empty defaults are omitted. State changes replace the current history entry so typing does not flood browser history, and `popstate` restores a visited URL. Theme remains a local preference, while popover visibility and scroll positions remain transient.

The visual system is grounded in four reviewed references: [Kong's 2026 brand guidelines](https://assets.prd.mktg.konghq.com/files/2026/02/6983c36f-kong-brand-2026.pdf) supply the exact Dark green `#000F06`, Electric lime `#CCFF00`, Bay `#B7BDB5`, White `#FFFFFF`, and ten-step neutral gray palette; Apple HIG contributes system-font legibility, succinct table rows, visible search scope, disclosure feedback, and responsive column reduction; Vercel Geist contributes the 13–14px utility type scale, semantic table treatment, tabular numeric columns, layered gray tokens, and low-elevation menu surfaces; VoidZero and Vite+ contribute crisp planes, thin structural borders, mono operational metadata, and compact radii. Brand colors map through semantic UI tokens rather than becoming decorative fields, so electric lime remains an accent instead of reducing information density or contrast.

The frontend is scaffolded from the bare TypeScript create-vue template. The application pins `vue` to 3.6.0-rc.2 so its Vapor runtime and compiler dependencies resolve from the same release. Every SFC opts into Vapor explicitly, and the application mounts with `createVaporApp`; the standard Vue plugin needs no separate global renderer switch. Lucide utility glyphs and Lobe provider marks are trusted build-time SVG compiled into one hidden symbol sprite; visible icons contain only an SVG `<use>` reference. Definition IDs are scoped to their symbol so gradients remain stable while repeated table rows share one path definition. Reviewed local assets cover marks that are missing or stale in Lobe: Microsoft Foundry uses Microsoft's current architecture icon, while Databricks uses the Simple Icons geometry in Databricks's official Lava 600 brand color. Provider marks never inherit interface text or state colors: multicolor marks retain their reviewed brand colors, monochrome marks use explicit black and white theme variants, and Cerebras retains its orange accent in both appearances. Kimi uses LobeHub's complete black monochrome mark on light surfaces and its color mark on dark surfaces, where the white body remains legible and the top-right element retains the official `#1783FF` blue. VitePlus remains the only dev, check, test, preview, and build tool entrypoint, with `vue-tsc` providing SFC type checking before production builds.

Virtual scrolling is implemented as a small framework-neutral range calculation plus Vapor-native table markup rather than a VDOM component dependency. The catalog's rows are deliberately fixed at 48px, so scroll offset, viewport height, and an eight-row overscan are sufficient to derive an exclusive render range and exact spacer heights. One small Vapor composable initializes OverlayScrollbars around explicit host/viewport pairs for the virtualized table, secondary filter popover, and model inspector. OverlayScrollbars owns only the restrained auto-hiding scrollbar chrome; Vue continues to own semantic content, native scroll behavior, and the table's range calculation. The semantic table publishes total and absolute row indexes for assistive technology. This keeps the entire application on the Vapor renderer while avoiding dynamic-row measurement complexity the product does not need.

The CSS begins with a modern reset, uses system fonts, and remains usable without JavaScript through an explanatory HTML shell. Vue interpolates only schema-validated catalog values and binds source URLs to normal link attributes; only trusted build-time icon SVG is injected as markup, and no external catalog value is treated as HTML.

## Automation and ownership

- GitHub Actions checks the project on pushes and pull requests.
- pnpm is the sole package manager. Its version is pinned in `package.json`, `pnpm-lock.yaml` is authoritative, and CI uses a frozen lockfile. Only the reviewed native tooling dependencies in `pnpm-workspace.yaml` may run install scripts.
- A scheduled workflow checks sources every 30 minutes, applies jitter, and commits only validated generated changes. Provider secrets are optional; absent secrets skip only their scoped inventories.
- The deploy workflow uses the repository’s pinned `void` dependency and GitHub OIDC; `void.json` declares a plain static `dist` deployment. `VOID_PROJECT` is the only repository variable.
- Renovate follows the repository’s Shanghai timezone, seven-day minimum release age, grouped automerge for non-major updates, and isolated manual major updates.
